how the whole system fits together.

jscc — the compiler

jscc is a pipeline of 5 stages:

stage 1 — lexer (lexer.cpp)

turns raw .jsc text into a flat list of tokens.

"def main() { return 0; }"
        ↓
[KW_DEF, IDENT(main), LPAREN, RPAREN, LBRACE, KW_RETURN, INT(0), SEMICOLON, RBRACE]

every token has a line and column number for error reporting.

stage 2 — parser (parser.cpp)

turns the token list into an Abstract Syntax Tree (AST).

[KW_DEF, IDENT(main), ...]
        ↓
Stmt::DEF {
  fn_name: "main"
  fn_body: [
    Stmt::RETURN { return_val: Expr::INT_LIT(0) }
  ]
}

the parser handles: functions, classes, sealed classes, generics, when expressions, async, decorators, type aliases, imports, packages.

stage 3 — import resolver (import_resolver.cpp)

resolves import jade.stdlib.io into actual symbols.

  • checks jade_modules/ first (local)
  • checks ~/.jade/registry/ second (global cache)
  • stdlib packages are registered directly in code
  • injects module symbols into the global scope

stage 4 — semantic analysis (sema.cpp)

type checks the AST and catches errors before codegen.

checks:

  • undefined variables → JD301
  • undefined functions → JD301
  • type mismatches → JD201
  • assigning to val → JD302
  • redeclaration → JD303
  • naming conventions (PascalCase classes, camelCase functions) → JD601/602
  • when arm bindings (injects pattern variables into scope)

errors use Rust-style output with file, line, column, underline, and help: hints.

stage 5 — codegen (codegen.cpp)

turns the AST into LLVM IR, then into native machine code.

AST
 ↓
LLVM IR (human-readable intermediate representation)
 ↓
native x86_64 / ARM64 / etc object code
 ↓
.joc file (multi-arch fat binary)

uses LLVM 18 as the backend. LLVM handles register allocation, instruction selection, and optimization. the codegen just builds the IR.

the .joc format

.joc is a fat binary with a custom header:

[4 bytes]   magic: "JADE"
[1 byte]    version: 1
[1 byte]    flags: bit0=has_bytecode, bit1=bytecode_only
[2 bytes]   reserved
[4 bytes]   num_slices
[4 bytes]   metadata_len
[N bytes]   metadata (package name)

slice directory (20 bytes per slice):
  [4 bytes]   arch tag: "x86_" | "ARM_" | "RV64" | "LLBC"
  [8 bytes]   offset
  [8 bytes]   size

slice data:
  [raw bytes] native object code or LLVM bitcode

a .joc always contains an LLBC (LLVM bitcode) slice by default. native slices are added with --target. on any machine, jvk picks the best slice — native if available, LLBC+JIT otherwise.

jvk — the Jade Virtual Kernel

startup sequence

1. parse flags (--jit-only, --native-only, --mem, etc)
2. JocLoader::load() — read .joc into memory, parse manifest
3. JocLoader::best_slice() — find native slice or fall back to LLBC
4. HAL::create() — create platform-specific HAL
5. Runtime::init() — set up VFS, scheduler, memory manager
6. run_native() or run_jit() — execute the program

HAL (Host Abstraction Layer)

hal.hpp defines a pure virtual interface. three implementations:

  • hal_linux.cpp — mmap, pthreads, epoll
  • hal_macos.cpp — mmap with MAP_JIT for Apple Silicon, pthreads, kqueue
  • hal_windows.cpp — VirtualAlloc, CreateThread, Win32 I/O

the HAL abstracts: memory allocation, executable memory mapping, threads, file I/O, time, process control.

native execution (run_native)

uses LLVM's RuntimeDyld:

  1. wraps the native slice bytes in an LLVM MemoryBuffer
  2. parses it as an ELF/COFF/MachO object file
  3. RuntimeDyld handles relocations
  4. a custom SectionMemoryManager resolves host symbols (printf, malloc, etc) via dlsym
  5. finds the main symbol and calls it

JIT execution (run_jit)

uses LLVM's MCJIT:

  1. parses the LLBC slice as LLVM bitcode
  2. creates an ExecutionEngine with MCJIT
  3. finalizeObject() compiles to native
  4. looks up main and calls it

JIT output is as fast as AOT native after the first run.

VFS (Virtual Filesystem)

apps see a virtual path like /data/config.json. the VFS resolves it to a real host path via mount points:

vfs.mount("/", "/home/user/myapp");
// /data/config.json → /home/user/myapp/data/config.json

scheduler

Scheduler wraps OS threads via the HAL. each Jade green thread maps to one OS thread. a real fiber/coroutine scheduler is planned for async support.

memory manager

MemoryManager currently delegates to the HAL's mmap. a proper slab allocator + the RC garbage collector from jade_stdlib will be wired in.

jade_stdlib — the standard library

written in C++ with C-linkage exports so jvk's RuntimeDyld can resolve them by name.

garbage collector (gc/gc.cpp)

implements Bacon-Rajan deferred reference counting with cycle detection:

  • every heap object has a GcHeader (rc, color, type_id, trace/destroy function pointers)
  • Rc<T> is the smart pointer — increment on copy, decrement on destroy
  • when rc hits 0 → immediate free
  • when rc > 0 but object looks like a cycle root → added to candidate buffer
  • when buffer fills → Bacon-Rajan mark-gray → scan → collect-white phases
  • cycle detector runs in the background, stops the world only during collection

string interpolation (src/interp.cpp)

#{var} is resolved at runtime via InterpolatedString:

  • compiler emits calls to jade_interp_literal() and jade_interp_str/int/float/bool()
  • jade_interp_build() concatenates everything into the final string

jpkg — the package manager

written in Python. three files:

  • jpkg.py — entry point, command dispatch
  • config.py — jade.toml parser, semver resolution
  • registry.py — GitHub registry client, download, publish

package resolution

jade.toml declares "jade.stdlib.http" = "^1.0.0"
        ↓
registry.py fetches packages/jade/stdlib/http/info.json from jade-pkgs
        ↓
finds versions satisfying ^1.0.0
        ↓
downloads to ~/.jade/registry/jade/stdlib/http/1.0.0/
        ↓
symlinks into jade_modules/jade.stdlib.http@1.0.0/

JadeUI

declarative UI framework. three layers:

Jade app code (.jsc)
        ↓
JadeUI widget tree (C++)
  Widget → Column → Row → Button → Text...
        ↓
IRenderer (abstract interface)
        ↓
SkiaRenderer (Skia GPU renderer)
        ↓
SDL2 OpenGL window
        ↓
OS window system

layout

inspired by Flutter's layout model:

  1. measure(constraints) — widget reports desired size
  2. layout(bounds) — widget places itself and its children
  3. paint(renderer) — widget draws itself

event handling

SDL2 events → translated to jade::ui::Event → bubbled through widget tree from leaf to root. first widget that returns true from handle_event consumes the event.
